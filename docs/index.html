1. Executive Summary

I analyzed the complete set of RAN1 session notes for agenda item 9.1 (AI/ML for NR Air Interface) across seven consecutive meetings, from #116 (Athens, Feb 2024) through #120 (Athens, Feb 2025). The analysis extracted every formal decision (agreement, conclusion, working assumption) and tracked Nokia and Ericsson's contributions, positions, and procedural roles throughout.

  • 237 formal decisions were reached across 7 sessions. Beam management leads in consensus maturity and is the most ready for production reference builds on Aerial today.
  • CSI prediction moved to specification support at RAN1 #120 (Feb 2025), with formal agreements now being made on framework reuse and resource types. Ericsson's multi-step prediction results (N4=4) are the strongest in the field and create the strongest compute case for GPU acceleration.
  • CSI compression converged on a Transformer backbone for Case 0 at #119, but three inter-vendor collaboration directions (A, B, C) remain open. Direction C would constrain every GPU implementation to a specific model architecture — this is the single highest-stakes open decision for Aerial's product roadmap.
  • Ericsson controls the positioning specification as moderator for agenda 9.1.2 across all 7 meetings. Any GPU-accelerated positioning implementation must track Ericsson's moderator summaries, as they define the exact spec text.
  • Nokia participates broadly but drives no sub-agenda. Nokia's differentiated approach — vehicular-focused evaluation assumptions, two-sided model emphasis — means they need a fundamentally different Aerial value proposition than Ericsson.
  • The RRC parameter surface defined in R1-2501143 (Qualcomm rapporteur document) maps directly to Aerial's pipeline: CSI-ReportConfig extensions → cuPHY, Associated ID → cuPHY-CP, OtherConfig → CU on Grace, LPP extensions → positioning NIM.

Nokia: The Flexibility Play

Nokia's 3GPP positioning tells a clear story: they're targeting diverse, mobility-heavy deployment scenarios (vehicular, in-car) and investing in two-sided models (CSI compression). This maps to a PIC strategy centered on:

  • ARC as a single-box edge platform — one GH200 at the cell site running cuPHY + cuMAC + NIM for D-RAN/C-RAN deployments where the DU is close to the road
  • Aerial Framework flexibility — Nokia's antenna-port-delay domain approach is unique among all 3GPP contributors; they need the Aerial Framework's open pipeline generation to implement their specific models
  • CSI compression Direction A/B/C tracking — Nokia's two-sided model focus makes the direction down-selection existential for their Aerial integration path
  • Live engagement: Nokia + T-Mobile field validation trials expected in 2026

Ericsson: The Compute Density Play

Ericsson's 3GPP positioning tells a different story: they control the positioning specification, dominate CSI prediction evaluation results, and push for compute-intensive multi-step prediction. This maps to:

  • ARC as a compute-dense C-RAN/Cloud-RAN platform — Ericsson's N4=4 multi-step prediction significantly increases inference demand vs N4=1 (multiple prediction steps per reporting interval); CPU-based L1 can't keep up
  • Positioning pipeline alignment — Ericsson's moderator summaries define the exact measurement formats (sample-based, Nt' from Nt consecutive samples, T=2k×Tc granularity); cuPHY must ingest these natively
  • pyAerial benchmarking — Ericsson's evaluation results are the most comprehensive in 3GPP; they'll want to reproduce them on GPU to validate Aerial's performance claims
  • Commercial context: Ericsson is an AI-RAN Alliance member; SoftBank (Ericsson RAN customer) planning commercial AI-RAN deployment in 2026
Same GPU platform, two different value propositions. The PIC for each partner needs to lead with the right story: flexibility for Nokia, compute density for Ericsson. The 3GPP record makes this concrete, not speculative.

3. Three Open Standards Decisions That Will Shape Aerial's Roadmap

Of the many open items across 5 AI/ML use cases, three decisions have outsized impact on how Aerial needs to evolve. These are the items I'd track as PIC and brief the cross-functional team on quarterly.

Decision 1: CSI Compression Direction Down-Selection

What's at stake: Three surviving directions for inter-vendor collaboration in CSI compression:

DirectionMechanismImpact on Aerial
A (Options 3a-1, 4-1)Offline parameter/dataset exchange between vendorsAerial preserves full flexibility — vendors keep proprietary models, exchange parameters offline
BDirect parameter transfer to UE over air interfaceModerate constraint — Aerial must support parameter encoding/transfer but model architecture remains flexible
CFully standardized reference modelMaximum constraint — the Transformer backbone agreed at #119 becomes mandatory; every GPU implementation must run exactly that model
If Direction C wins, cuPHY needs a TensorRT-integrated inference stage that can load and execute a standardized Transformer model as a first-class pipeline component. This is not a sidecar — it's core L1 functionality. If Direction A wins, the Aerial Framework's open pipeline generation becomes the competitive advantage. The down-selection will likely happen in the next 2–3 RAN1 meetings.

Decision 2: CSI Processing Timeline for AI/ML

What's at stake: The CSI processing timeline for BM-Case1 and BM-Case2 is still open. This determines how much latency budget exists for AI inference within the L1 processing window.

This is where GPU parallel processing becomes the differentiator. A tight timeline favors GPU (more complex models in the same time budget via parallelism) over CPU. NVIDIA's interest is in a timeline that accommodates AI inference without forcing a choice between model complexity and latency. Both Nokia (needs complex models to compensate for generalization gaps) and Ericsson (needs time for multi-step prediction) would benefit from a generous timeline.

Decision 3: Model Transfer Scalability (Case z4)

What's at stake: Case z4 (known model structure + parameter transfer OTA) is the surviving model transfer mechanism. How model parameters are encoded and how the UE signals readiness directly impacts GPU-accelerated model lifecycle management.

The ARC platform can manage model lifecycle (training, compression, transfer) as a containerized NIM function. But the standard needs to define parameter encoding formats that scale across antenna configurations and bandwidths without creating air interface bottlenecks. This is a topic where NVIDIA has a legitimate technical voice in 3GPP-adjacent discussions.

6. Five Inputs I'd Bring to the Aerial Product Team

The DevGAM role sits at the intersection of partner engineering and NVIDIA product teams. Here are five observations from the 3GPP analysis that I'd escalate as product strategy inputs:

1. cuPHY needs a standardized model inference path. If CSI compression Direction C wins, the Transformer backbone becomes mandatory. cuPHY needs a TensorRT-integrated inference stage — not a sidecar, a first-class pipeline component. Timeline: ready before Rel-19 freeze.
2. pyAerial/Sionna should ship with 3GPP evaluation profiles. The 3GPP evaluation framework specifies exact conditions (30/60 km/h, 5/20ms CSI-RS periodicity, indoor/outdoor distributions). Pre-configured profiles matching TR 38.843 would let NEP engineers validate AI models against 3GPP assumptions in one line of Python. This is a low-cost, high-differentiation feature.
3. A Positioning NIM is the cleanest new NIM opportunity. AI/ML positioning is a well-defined inference problem: 3GPP-formatted measurements in, position estimate out. The measurement formats are nearly finalized (Ericsson has been driving them as moderator). Package LMF as a NIM on ARC.
4. Associated ID needs lifecycle management in the SDK. The Associated ID mechanism bridges training and inference across all use cases. It's not a simple parameter — it's a lifecycle spanning data collection, training, inference configuration, and performance monitoring. This needs first-class support in cuPHY-CP and the L2 stack.
5. NEPs need a "what's standardized vs. what's yours" map. The biggest adoption barrier I'd anticipate is confusion about where 3GPP's interface specification ends and Aerial's implementation freedom begins. A clear mapping document per use case becomes the foundation of every PIC engagement. Aerial's value proposition is clearest in the implementation-specific zone.

8. Positions I'd Advocate in the AI-RAN Alliance and 3GPP Context

Three positions I'd advocate in the AI-RAN Alliance and 3GPP-adjacent forums, each grounded in specific 3GPP evidence:

Position 1: "The Standard Creates a GPU-Shaped Opportunity"

3GPP standardizes the measurement interfaces and signaling procedures but explicitly leaves model architecture as implementation-specific for NW-sided and UE-sided models. The gNB decides what AI to run. This is where GPU acceleration has maximum impact: behind a standards-compliant interface, running models that are too compute-intensive for CPU-based L1.

The evidence is specific: Ericsson's N4=4 multi-step CSI prediction — the strongest results in 3GPP — substantially increases inference demand by requiring multiple prediction steps per reporting interval. NW-sided positioning inference across Cases 3a/3b runs on the gNB. These aren't hypothetical GPU workloads; they're what the standard is specifying right now.

Position 2: "Processing Timelines Should Accommodate AI Inference"

The CSI processing timeline for AI/ML is still open in 3GPP. NVIDIA's interest is in a timeline that doesn't force a trade-off between model complexity and latency. Both Nokia (higher generalization degradation means more complex compensating models) and Ericsson (multi-step prediction needs inference time) align with this position.

Position 3: "Model Transfer Must Scale for Commercial Deployment"

Case z4 parameter transfer needs efficient encoding formats that scale across antenna configurations and bandwidths. ARC can manage model lifecycle as a NIM, but the standard's parameter format determines whether this scales or bottlenecks the air interface.
